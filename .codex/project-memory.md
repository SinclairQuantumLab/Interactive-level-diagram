# Project Memory

## What This Project Is

- This is an interactive atomic level diagram app aimed at novice users.
- The core product goal is local, approachable, visual exploration of atomic structure.
- The app is not meant to force end users into command-line workflows.
- The browser experience is the only supported runtime target.

## Primary User Expectations

- Users should be able to edit diagram data files directly in-place.
- Users should be able to understand the app mostly from the UI itself.
- Diagram structure, references, notes, labels, and measurements should feel visually integrated into the dashboard rather than managed through detached admin-like tools.
- Scientific notation, uncertainties, references, and spectroscopic labels should be displayed in a way that respects physics conventions, not generic software defaults.

## Runtime Model

### Browser mode

- This is the main mode.
- Users choose the `diagrams/` folder through the browser folder-access flow.
- The app tries to make that folder choice obvious and instructive.
- The browser remembers the folder handle once granted.
- Direct automatic access to `./diagrams` is not possible in a normal browser page because of the browser security model.

## Durable UX Decisions

- The home panel is intentionally simplified and should not carry tutorial clutter.
- Global help/how-to-use text belongs in the `?` help panel, not scattered around the home panel.
- References belong in a dedicated references panel plus lightweight hover/pinned reference panels.
- Hide/show and measurement management should happen visually on the diagram itself whenever practical.
- State and transition interactions should remain directly manipulable from the rendered diagram, not hidden in secondary menus.
- Inline TeX-like math should work broadly across dashboard text, not only in one rendering path.

## Diagram Data Schema

The project intentionally moved away from TOML and now uses YAML only.

### Top-level structure

- `meta`
- `species`
- `states`
- `transitions`

### Species

Current expectations:

- `species`
- `atomic_mass_number`
- `ionization_number`
- `nuclear_spin`
- `notes`

Important historical decision:

- Species display notes were renamed away from `properties`.
- `species.notes` is the intended place for display-oriented species notes.
- `nuclear_spin` remains a real physics field, not just display text.

### States

Current expected state fields:

- `id`
- `notation`
- `j`
- `energy`
- `hyperfine_constants`
- optional `notes`
- optional `hyperfine`

Important rules:

- `notation` is the display label.
- `id` is the stable internal reference.
- Old fallbacks like `label` or `term` should not be reintroduced.
- Old `detail_note` style text was intentionally removed in favor of concise `notes`.
- `notes` is a list, not a singular `note`.

### Hyperfine constants and energies

- Units live in the value string, not in the field name.
- The app parses strings like:
  - `384.230352862(5) THz\cite{Key}`
  - `84.72(?) MHz\cite{Key}`
- The app is responsible for parsing:
  - central numeric value
  - uncertainty notation
  - unit
  - inline citations
- Measured scientific values are intentionally treated as authored display strings that also carry computable meaning.

### Citations

- Inline citation form is `\cite{BibKey}`.
- This is preferred over separate per-field reference sections for authored content.
- Notes and measured-value fields may include inline citations.
- The app strips citation text from most normal value display and renders citations as superscripts where appropriate.

### Transitions

Transitions are undirected.

Current expected transition fields:

- `between: [stateA, stateB]`
- optional `type`
- optional `frequency`
- optional `strength`
- optional `notes`
- optional `show`

Important rules:

- `between` is the identity source.
- Endpoint order must not matter.
- Old alternatives like `from` / `to` or `state_pair` are intentionally deprecated and should stay gone.
- Transition-specific persistent UI state should also serialize in `between` form, not old internal string ids.

### Endpoint notation

Current endpoint notation conventions:

- fine: `5P3/2`
- hyperfine: `5P3/2[F=3]`
- zeeman: `5P3/2[F=3,mF=2]`

This same notation should be used consistently across:

- transition endpoints
- hidden transition serialization
- measurement serialization
- pinned or selection references where possible

### Measurements in copied layout YAML

- Measurements should serialize with `between: [...]`.
- Old internal measurement id strings like `fine:5D3/2 <-> fine:5P3/2` were intentionally removed from the exported schema.

## Bibliography and References

- Each diagram YAML may have a same-basename `.bib` file.
- References are rendered simply, APS-style, and with clickable links when URLs are present.
- Hover reference panels should stay visually above all other panels.
- Left-clicking citation superscripts should open the associated URL.
- Right-clicking citation superscripts should pin the reference panel.
- The big references panel is controlled by the references toggle.
- Pinned reference panels should continue to exist independently of whether the bibliography panel itself is shown.

## Numbers, Uncertainties, and Formatting

- The app parses short-form uncertainty notation such as:
  - `1.53(3)`
  - `6.287(32)e+6`
  - `1.43(+2,-5)e-3`
- Dashboard formatting groups digits with:
  - commas in the integer part
  - spaces every three digits in the fractional part
- When a decimal limiter is applied to a rendered label, the uncertainty suffix is intentionally dropped from that rendered label.
- If a value truly lacks a known uncertainty, `(?)` is used explicitly.

## Physics and Visualization Conventions

### Fine, hyperfine, and Zeeman structure

- Fine-state energies are internally handled in THz.
- The diagram config itself should also provide THz-based energies; the app should not carry unnecessary unit-conversion compatibility.
- Hyperfine constants use units written in the values, typically MHz.
- Zeeman controls are split into:
  - actual magnetic field in G with a linear slider and min/max inputs
  - a separate visualization-scale slider

### Hyperfine and Zeeman panel content

Current intent:

- Panels should be compact and not padded with explanatory prose unless the config provides actual notes.
- Rows like redundant notation, manifold, J, and action were intentionally removed.
- `g_F` and `m_F` should stay in plain-font style with plain subscripts, not math-font over-styling.
- Lowest hyperfine states should not redundantly show `interval below` or `from lowest`.

### Transition visualization

- Transition color is estimated from frequency or wavelength.
- Visible-range transitions use rough visible spectral colors.
- UV and IR use softened violet and red-like colors.
- Lines are currently straight, not curved.
- Anchor slotting tries to reduce obvious crossings.
- Wavelength labels try to avoid collisions and may shift left or right when crowded.
- Users can choose which transition panel items appear on the on-diagram label.
- If no items are selected, no transition label should be shown.

### Measurement visualization

- Multiple measurement lines are allowed.
- Measurement labels are intentionally more muted than transition labels.
- Measurements may be removed from the diagram itself when the measure tool is active.
- Measurement panels support inline note editing and label-field selection.
- Measurement labels should not fall back to a hidden default once the visible frequency item is unchecked.

### Hide/show behavior

- Hide mode grays out hidden items while the tool is active.
- Outside hide mode, hidden items are removed from normal view.
- Hidden states should remove associated transitions and measures from the normal dashboard view.
- Hyperfine hidden runs should collapse edge space only; visible hyperfine states should not be rescaled against each other.
- Zeeman hidden runs should collapse to a single placeholder slot with explicit three-dot markers.
- Zeeman alignment should stay centered around `m_F = 0` when possible, including hidden-run placeholders that contain `m_F = 0`.

## Rendering and Layout Notes

- Fine-state labels live in an HTML overlay layer rather than pure SVG text.
- That overlay must stay synchronized with the diagram transform and scale.
- Earlier bugs came from treating panels or labels in mixed coordinate systems; the current rendering path relies on the full SVG screen transform where needed.
- The fit-view logic is preferred over a fixed default camera.
- `Reset View` was intentionally renamed to `Fit View`.
- Hidden-state ellipsis markers are rendered as explicit SVG dots, not Unicode glyphs.

## Persistence and Layout State

- Layout copy/load is YAML-only.
- Exported layout includes app version.
- Persistent state versioning in app storage was deliberately bumped when schema cleanup removed older compatibility paths.
- Undo/redo should not treat pan/zoom as history steps.
- Slider drags should collapse into one undoable change from drag start to drag end.

## Versioning and Workflow

- GitHub Flow is now the intended repo workflow.
- Branch from `main`.
- Keep short-lived feature and fix branches.
- Merge back into `main`.

App versioning rules:

- On `main`: `YYYYMMDD.d`
- On non-`main` branches: `YYYYMMDD.<branchname>.<commit hash>`

Implementation notes:

- The displayed version is overridden at runtime by `src/generated-app-version.js`.
- That file is generated and ignored by Git.
- Repo-local hooks in `.githooks/` refresh that file after commits, checkouts, merges, and rewrites.
- Git does not store `core.hooksPath` in the repo itself, so a fresh clone on another machine must set it again locally:
  - `git config core.hooksPath .githooks`

## Things To Preserve

- Browser-first direction.
- YAML-only schema.
- `between: [...]` identity style.
- No unnecessary compatibility layers.
- Integrated on-diagram interactions for hide/show and measurement management.
- Simple APS-style references.
- TeX-like notation support across dashboard text, not just one label path.

## Things A Future Codex Should Check Before Major Refactors

- Whether a change breaks direct browser use.
- Whether a change reintroduces old schema fallbacks.
- Whether a change makes novice use more terminal-dependent.
- Whether a change splits one visual behavior across multiple duplicated layout pipelines.
- Whether a change breaks the consistency between rendered identifiers and exported identifiers.
